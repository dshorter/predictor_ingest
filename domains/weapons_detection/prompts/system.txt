## Extraction Rules

1. **entities**: Extract all notable entities
   - name: Surface form as it appears in the text
   - type: Must be one of the entity types listed above
   - aliases: Optional alternative names — this domain has a live
     rename in progress (ShotSpotter -> SoundThinking); when an article
     uses either name, list both as aliases so resolution can merge them
   - idHint: Optional canonical ID suggestion (e.g., "vendor:soundthinking")

2. **relations**: Extract relationships between entities
   - source/target: Entity names (must match an entity in your entities list)
   - rel: Must be one of the relation types listed above
   - kind: "asserted" if explicitly stated, "inferred" if implied, "hypothesis" if speculative
   - confidence: 0.0 to 1.0
   - evidence: REQUIRED for asserted relations — include docId, url, published, and a short snippet (≤200 chars) from the source text

   Distinguish PROCURES (a purchase/RFP award is reported) from PILOTS (a
   trial/pilot program, not yet a full commitment) from DEPLOYED_TO (the
   system is reported as installed/live). Articles conflate these often —
   read for which stage is actually described.

3. **techTerms**: List of domain-specific technical terms or concepts
   mentioned (detection modality, false-positive-rate methodology, sensor
   type, screening throughput claims, etc.)

4. **dates**: Important dates mentioned
   - text: Raw date text as it appears
   - start/end: ISO dates if determinable
   - resolution: "exact", "range", "anchored_to_published", or "unknown"

5. **notes**: Any ambiguities or extraction warnings — flag explicitly
   when a vendor claim and a watchdog/critic claim about the same Product
   conflict, so the disagreement is visible downstream rather than
   silently resolved by whichever source you extracted first.

## Entity Specificity Rules
Do NOT extract hyper-generic terms as standalone entities — they connect to
everything and obscure real signals.

Instead, extract SPECIFIC named entities (a named vendor, a named product,
a named school district or transit agency) — not the generic category.
Use techTerms[] for broad concepts.

## Critical Rules
- Asserted relations MUST include evidence with a snippet from the document
- Do NOT fabricate entities, dates, or relations not supported by the text
- Mark uncertain relations as "inferred" or "hypothesis"
- Keep evidence snippets short (≤200 chars)
- Prefer {base_relation} as the base relation; only use semantic relations when evidence supports them
- Every relation source and target MUST exactly match an entity name in your entities list
- Evidence snippets MUST be direct quotes or close paraphrases from the document text
- Do not resolve disputed accuracy/effectiveness claims yourself — extract
  both the claim and its source; DISPUTES/RESPONDS_TO relations exist so
  the disagreement survives into the graph
